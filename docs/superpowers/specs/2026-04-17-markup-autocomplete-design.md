# Design: Markup Editor Autocomplete

Autocomplete for the page-edit markup editor: tag names, attribute names, and attribute values (enum + boolean), with localized titles pulled from the JSON Schema.

## Context

The markup editing mode shipped in commits `2fd7ae7f` (first draft) and `f5597dd7` (CodeMirror integration). Users can toggle between the VJSF form editor and a pseudo-HTML markup editor; both modes bind to the same `draftConfig.elements` model. The architecture is documented in `docs/architecture/page-edition-markup-language.md`.

Today the CodeMirror editor provides syntax highlighting, folding, indentation, and inline lint diagnostics from the deserializer — but **no autocomplete**. Users have no in-editor way to discover which tags exist, which attributes each tag accepts, or which enum values are valid. This is the single biggest usability gap for manual editing of non-trivial pages.

## Goal

Add three-level autocomplete to the markup editor:

- **Tag names** — typing `<` suggests all element types and virtual tags (`tab`, `panel`, `left`, `right`, `action`, `filters`).
- **Attribute names** — inside an open tag, suggests the attributes of that tag, including dot-notation attributes (`background.color`).
- **Attribute values** — inside attribute-value quotes, suggests enum values for enum attributes, or `true`/`false` for booleans.

Each suggestion displays a short **localized title** sourced from the JSON Schema (matching what the form editor shows), in the current UI locale with English fallback.

Out of scope for this spec:
- Context-aware tag filtering (e.g., only suggesting `<tab>` inside `<tabs>`). The v1 tag list is flat. This can be layered on later without architectural changes.
- Long descriptions / docs panels. We show titles only; `description` fields stay unused.
- JSON Schema validation with source-map diagnostics (separate future work — see architecture doc §4.3).
- AI integration.

## Design Principles

- **Descriptors stay the source of truth.** All autocomplete data comes from the existing build-time descriptor pipeline. No runtime schema access, no duplicated metadata.
- **Locale resolution lives in one place.** A single helper handles `locale → en → raw name` fallback for tags, attributes, and enum values alike.
- **Stateless extensions.** The CodeMirror extension captures the locale at mount time. Locale changes take effect on the next editor mount (Markup mode toggles already unmount/remount the editor).

## Architecture Overview

Four areas change, plus tests.

### Build time — `build/markup/`

Extend the schema analyzer (`schema-analyzer.ts` and/or its callers in `generate-grammar.ts`) to also extract localized titles while walking the schemas. Titles are attached to:

- each real element type (keyed by `type.const`)
- each attribute (direct, dot-notation leaves, and item attributes on virtual-tag slots)
- each virtual tag slot (`left`, `right`, `tab`, `panel`, `action`, `filters`, etc.)
- each enum-value choice, when the schema expresses enums as `oneOf: [{ const, title }]`

Title sources in the schema (in priority order):
1. `x-i18n-title-<locale>` sibling fields (e.g. `x-i18n-title-fr`) — one entry per locale
2. `title` field — recorded under the `en` key unless indicated otherwise

The analyzer emits `titles` as `Record<locale, string>` only when at least one title was found.

### Runtime — `shared/markup/`

**`types.ts`** — additive fields on the three descriptor types:

```typescript
export interface TagDescriptor {
  tagName: string
  contentProperty: string | null
  childrenSlots: ChildrenSlot[]
  attributes: AttributeDescriptor[]
  hiddenProperties: string[]
  titles?: Record<string, string>
}

export interface ChildrenSlot {
  property: string
  virtualTag: string | null
  itemAttributes?: AttributeDescriptor[]
  kind: 'direct' | 'structured' | 'link'
  titles?: Record<string, string>
}

export interface AttributeDescriptor {
  name: string
  jsonPath: string[]
  type: 'string' | 'number' | 'integer' | 'boolean' | 'string-array'
  enumValues?: (string | number)[]
  required: boolean
  default?: unknown
  titles?: Record<string, string>
  enumTitles?: Record<string, Record<string, string>>  // value-as-string → locale → title
}
```

**`tag-descriptors.ts`** is regenerated by the build step to carry the new fields.

**New file `codemirror/completion.ts`** exports:

```typescript
export function portalMarkupCompletion (locale: string): CompletionSource
```

**`codemirror/index.ts`** — `portalMarkupExtensions` grows a `locale` option:

```typescript
export function portalMarkupExtensions (opts: { locale: string }): Extension[] {
  return [
    portalMarkup(),
    autocompletion({ override: [portalMarkupCompletion(opts.locale)] }),
    portalMarkupLinter
  ]
}
```

### UI — `ui/src/components/page-edit/page-edit-elements-markup.vue`

- Import `useI18n()` and read `locale.value` at mount time.
- Pass `{ locale: locale.value }` into `portalMarkupExtensions`.
- Add `...completionKeymap` to the existing `keymap.of([...])` call in `buildExtensions` so explicit completion key bindings (Ctrl-Space, Enter, Esc, etc.) take effect.
- No reactive re-wiring: locale changes mid-session are not propagated (accepted trade-off — the editor remounts whenever the user toggles between Form and Markup modes).

### Tests — `tests/features/markup/`

New file `completion.unit.spec.ts` with one test per completion branch. Existing analyzer tests gain one assertion that `titles` is populated for a representative element.

## Completion Source Logic

`portalMarkupCompletion(locale)` returns a `CompletionSource` that inspects the Lezer syntax tree at the cursor and branches on node type.

### Context detection

Use `syntaxTree(state).resolveInner(pos, -1)` and walk parents to decide which of the three contexts the cursor is in.

| Cursor position | How it's detected | Output |
|---|---|---|
| Inside `TagName` / right after `<` | Innermost node is `TagName`, or prev char is `<` | Tag names (real + virtual) |
| Inside `AttributeName` or whitespace after `TagName` | Enclosing `OpenTag`/`SelfClosingTag`, cursor not inside a value | Attributes of that tag |
| Inside `AttributeValue` (between the quotes) | Enclosing `AttributeValue`, and its sibling `AttributeName` is known | Enum / boolean values |

The source returns `null` when none of these contexts apply or when the enclosing tag is unknown (so nothing is suggested on garbage input).

### Tag name branch

Emit one `Completion` per entry in `tagDescriptors`:

```typescript
{ label: 'title', detail: localized(desc.titles, locale), type: 'class' }
```

Plus one entry per unique virtual tag, collected by walking every `TagDescriptor.childrenSlots` and taking the unique `{ virtualTag, titles }` pairs. Virtual tags use `type: 'keyword'` so CodeMirror renders a visibly different icon.

`from` is set to the start of the partial token; `validFor: /^[A-Za-z0-9_-]*$/`.

### Attribute name branch

1. Walk up to the enclosing `OpenTag` / `SelfClosingTag`, read the `TagName` text.
2. Look up the descriptor:
   - If it matches a real tag: use `descriptor.attributes`.
   - Otherwise, scan every `TagDescriptor.childrenSlots` for a slot with matching `virtualTag` and use that slot's `itemAttributes`.
   - If nothing matches: return `null`.
3. Emit one `Completion` per attribute:

```typescript
{
  label: attr.name,
  detail: localized(attr.titles, locale),
  type: 'property',
  info: attr.required ? 'required' : undefined,
  apply: (view, _c, from, to) => {
    // Insert `name=""` and place cursor between the quotes.
    view.dispatch({
      changes: { from, to, insert: `${attr.name}=""` },
      selection: { anchor: from + attr.name.length + 2 }
    })
  }
}
```

`validFor: /^[A-Za-z0-9_.\-]*$/` (dot included for dot-notation attributes).

### Attribute value branch

1. Walk up to find the enclosing `Attribute` node (the `AttributeValue`'s parent).
2. Read the `AttributeName` text from its first child.
3. Resolve the descriptor on the enclosing tag (same lookup as the attribute-name branch).
4. Find the `AttributeDescriptor` whose `name` matches. If not found: return `null`.
5. Emit suggestions based on type:
   - `enumValues` present: one entry per value, `label: String(value)`, `detail: localized(attr.enumTitles?.[String(value)], locale)`, `type: 'enum'`.
   - `type === 'boolean'`: `true` and `false`.
   - Otherwise: return `null`.

`from`/`to` are set to the positions just inside the quotes so completion replaces only the current value.

### Locale resolution helper

```typescript
function localized (titles: Record<string, string> | undefined, locale: string): string | undefined {
  if (!titles) return undefined
  return titles[locale] ?? titles.en ?? Object.values(titles)[0]
}
```

Single helper reused across the three branches.

### Fallback behavior

- Unknown tag in an attribute context → return `null` (no wrong suggestions).
- Partial Lezer parse (user typing, broken markup) → `resolveInner` still returns the best-fit node; our branches handle missing parent links by returning `null`.
- No titles for a locale → English fallback → raw name fallback (never undefined).
- `portalMarkupExtensions` bundles `autocompletion(...)`, but the Vue component must also include `completionKeymap` (from `@codemirror/autocomplete`) in its `keymap.of([...])` call so Ctrl-Space / Enter-to-accept / Esc-to-dismiss work. Auto-popup on typing works without the keymap addition.

## Locale Wiring

### Source

`useI18n().locale` in `page-edit-elements-markup.vue` — same ref driving the surrounding form editor's labels.

### Threading

```typescript
// page-edit-elements-markup.vue
const { locale } = useI18n()
// inside onMounted, before creating the EditorView:
const extensions = buildExtensions(locale.value)
```

`buildExtensions` takes the locale and passes it through to `portalMarkupExtensions({ locale })`. No other caller of `portalMarkupExtensions` exists, so the signature change is local.

### Reactivity

None. The editor captures the locale at mount. If the user toggles the portal UI language mid-session, completions keep the mount-time locale until the next Form↔Markup toggle forces a remount. This matches the session-stable nature of the portal's UI locale and keeps the descriptors stateless.

## Testing Strategy

### Unit tests — `tests/features/markup/completion.unit.spec.ts`

Helper:

```typescript
function completionsAt (doc: string, cursor: number, locale = 'en'): readonly Completion[] {
  const state = EditorState.create({
    doc,
    extensions: [portalMarkupLanguage]
  })
  const source = portalMarkupCompletion(locale)
  const ctx = new CompletionContext(state, cursor, false)
  const result = source(ctx)
  return result?.options ?? []
}
```

Cases:

1. **Tag name** — `<tit|` → options include `title` with localized `detail`; virtual tags `tab`, `left`, `action` are present.
2. **Attribute name on a real tag** — `<title |` → includes `titleSize`, `centered`, `line.position`; does not include attributes from other tags.
3. **Attribute name on a virtual tag** — `<tabs>\n  <tab |\n</tabs>` → includes the `tab` slot's `itemAttributes` (`title`, `icon`).
4. **Enum value** — `<title titleSize="|"` → options are the `titleSize` enum (`h1`, `h2`, …), with localized `detail` where available.
5. **Boolean value** — `<title centered="|"` → exactly `true` and `false`.
6. **Unknown tag** — `<nonsense |` → source returns `null` (no attribute suggestions).
7. **Locale fallback** — repeat case (1) with `locale='fr'`; assert French titles where present, English otherwise, raw names where nothing exists.

### Build-time tests

One additional assertion in the existing analyzer unit tests: for a representative element (e.g. `banner`), the generated descriptor has `titles.fr` and `titles.en` populated, and at least one of its attributes has `titles` populated.

### Manual verification

After `npm run build-types` succeeds and unit tests pass, start the dev UI (`bash dev/status.sh` to confirm services are up), open a page editor, toggle to Markup mode. Then:

1. Type `<` — a tag list appears with localized labels matching the current UI locale.
2. Pick a tag, type a space — attribute list for that tag appears.
3. Type an enum-valued attribute name, `=` — the insertion helper places the cursor inside empty quotes and a value list appears.
4. Type a partial value — the list narrows.
5. Switch the portal UI language, toggle Form → Markup again, repeat (1) — labels appear in the new locale.

## Risks and Trade-offs

- **Flat tag list.** Suggesting `<tab>` at the root of the document is semantically wrong, but it's cheap and makes the scope self-contained. Users who try it will see a lint error from the deserializer (virtual tag without a valid parent). Context-aware filtering is a clean future addition.
- **Schema without titles.** If a schema entry carries no `title`, its completion has no `detail`. This is visible (empty right side) but not misleading. Fix is to add titles to the schema, not to the completion source.
- **Locale snapshot.** Mid-session locale swaps produce briefly stale completion labels. Markup-mode toggles or page reloads resolve it. A reactive facet could be added later if this becomes a real complaint.
- **Insertion helper (`name=""`).** If the user already typed part of an attribute name and accepts a completion, the helper replaces `from..to` with `name=""`. We rely on CodeMirror's `from` being set to the start of the partial token so this works correctly for in-progress typing.

## Verification Checklist

1. `npm run build-types` succeeds; generated `tag-descriptors.ts` includes `titles` fields for at least one representative element.
2. `npm run test -- tests/features/markup/completion.unit.spec.ts` passes all seven cases.
3. `npm run test -- tests/features/markup/` continues to pass (existing deserializer/serializer/analyzer tests untouched).
4. Manual steps 1–5 under "Manual verification" observed in the browser.
